## Plan: 排序与信誉评分

把两个目标拆开实现：先修正商品列表的排序选择器和排序默认行为，确保距离、价格升序、价格降序都能由用户显式选择；再补上购买后的卖家评分链路，并把个人中心的“被点赞数”和“诚信分”改成真实统计值。为了避免前端假数据，点赞统计会走服务端持久化，诚信分会基于订单评分聚合计算，统一在后端输出给个人中心。

**Steps**
1. 先梳理并改正商品列表的排序入口，集中处理 `web/src/components/tabs.tsx` 里的排序下拉和 `web/src/App.tsx` 里的 `filteredMarket` 逻辑。
2. 更新排序状态的默认值和选项文案，让“默认 / 距离由近到远 / 价格由低到高 / 价格由高到低 / 最新发布”都成为明确可选项；只有在用户显式选中距离排序时才按定位距离排序，避免页面始终表现成“近的靠前”。
3. 评估是否需要把排序逻辑抽成一个小型排序函数，保证 `filteredMarket` 只负责过滤和调用排序，不再混入其他业务分支。这个步骤依赖第 1 步。
4. 为购买后的卖家评价补数据模型：在 `server/src/db.ts` 和 `shared/src/index.ts` / `web/src/types.ts` 中补订单评分字段，建议直接把 1-10 星评分存到订单记录上，方便一单一评、后续聚合卖家诚信分。
5. 在 `server/src/routes.ts` 新增订单评分接口，限定只有该订单的买家才能评分，且只允许完成交易后的订单评分；评分成功后返回更新后的订单数据，供前端刷新个人中心和订单列表。这个步骤依赖第 4 步。
6. 在 `web/src/components/modals.tsx` 增加一个评分弹框，沿用现有 modal 样式，提供 1-10 星选择和提交入口；在 `web/src/App.tsx` 的购买成功回调里接入这个弹框，并保留一个“稍后再评”的退路，避免用户关闭弹框后无法补评。这个步骤依赖第 4、5 步。
7. 把个人中心的“诚信分”改成真实值：后端按卖家收到的订单评分平均值计算，输出 0-10 的小数；前端在 `PostProfileTab` 里替换掉当前硬编码的 `98`。这个步骤依赖第 4 步。
8. 把个人中心的“被点赞数”改成真实统计：新增服务端持久化的点赞/收藏关系表，把现有收藏行为同步到后端，再按卖家商品被收藏的次数聚合出统计值并返回给个人中心。这个步骤依赖第 1 步和第 4 步，且会连带更新当前收藏切换流程。
9. 在 `web/src/components/tabs.tsx` 的个人中心订单列表里补一个“去评分 / 已评分”状态展示，让评分弹框不是唯一入口，避免购买后错过一次就没有补救路径。这个步骤依赖第 6 步。
10. 最后统一调整类型和接口封装，保证 `web/src/services/api.ts`、`shared/src/index.ts`、`web/src/types.ts`、以及 `server/src/routes.ts` 的返回结构一致，避免前后端因为新增字段产生编译或运行时不匹配。

**Relevant files**
- `web/src/App.tsx` — 排序状态、`filteredMarket`、购买成功后的评分弹框触发点、个人中心数据刷新
- `web/src/components/tabs.tsx` — 商品广场排序下拉、个人中心统计卡片、订单列表里的评分入口
- `web/src/components/modals.tsx` — 新增评分弹框，复用现有模态层样式
- `web/src/services/api.ts` — 新增订单评分和点赞统计相关 API 调用封装
- `server/src/routes.ts` — 购买后评分接口、个人中心统计数据组装、点赞同步接口
- `server/src/db.ts` — 订单评分字段、点赞关系表或相关索引的建表迁移
- `shared/src/index.ts` — 共享类型补充订单评分与个人统计字段
- `web/src/types.ts` — 前端类型同步，避免现有页面组件编译失败

**Verification**
1. 先做最小行为验证：打开商品广场确认排序下拉能在距离、价格升序、价格降序、最新发布之间切换，且未选择距离排序时不会被定位强制覆盖。
2. 启动后手动走一遍购买流程，确认下单成功后会弹出卖家评分框，提交 1-10 星后个人中心订单列表能显示“已评分”。
3. 检查个人中心数据是否从真实统计返回：诚信分随订单评分变化，被点赞数随收藏关系变化，而不是固定常量。
4. 跑一次前后端类型/构建检查，重点确认 `shared`、`web`、`server` 的新增字段不会造成类型不一致或 API 返回缺字段。
5. 如果项目里有现成的测试或 lint 命令，再对新增接口和新增组件做一次定点验证；如果没有，就至少用页面联调和后端接口调用确认行为闭环。

**Decisions**
- 诚信分按卖家收到的 1-10 星评分平均值展示，保留小数，直接落在 0-10 区间。
- 被点赞数按其他用户收藏卖家商品的次数统计，不做前端假值。
- 排序保持前端本地完成，不额外增加服务端排序参数。
- 评分以订单为单位，一单一评；购买成功后立即弹框，同时保留后续补评分入口。
- 后端改动保持最小化，只补必要的数据字段、统计接口和评分接口，不重构现有购买/商品主流程。

**Further Considerations**
1. 是否需要把“收藏”完全改成服务端同步而不是保留 localStorage 版本？推荐同步到服务端，否则“被点赞数”无法跨设备、跨用户统计。
2. 是否要在个人中心展示“诚信分”的明细来源，例如最近评分、评分总数？如果要，后端可以顺手返回 `ratingCount` 一起展示。
